# Standard — UI Test IDs & Accessibility (Public)

Status: ACTIVE  
Scope: Public standard (no internal endpoints, no secrets)

## 0. Purpose

Enable deterministic UI automation (agents + CI) without brittle selectors.

This standard ensures:
- Agents can reliably locate UI elements
- UI assertions are stable across refactors
- Accessibility is improved as a side effect

## 1. Required Attributes

### 1.1 `data-testid`
Must exist on automation-critical elements:
- Primary CTA buttons
- Form inputs and submit buttons
- Error banners/toasts
- Success confirmations
- Loading states that affect flow completion

### 1.2 Accessible name (a11y)
Interactive controls must have a stable accessible name via:
- `aria-label`, or
- associated `<label>`, or
- semantic element text that is stable and not dynamic

## 2. Naming Convention

### 2.1 Format
Use dot-delimited semantic identifiers:

`<surface>.<screen>.<component>.<action|state>`

Examples:
- `gateway.intake.prompt.submit`
- `auth.login.form.submit`
- `order.checkout.payment.confirm`
- `order.status.banner.error`

### 2.2 Rules
- Must be stable; never include dynamic values (IDs, timestamps, user names)
- Prefer lower-case with dots, no spaces
- Reuse across platforms where possible (web/mobile parity)

## 3. Selector Policy (Automation)

### 3.1 Preferred selector order
1) Accessibility queries (role + name)
2) `data-testid`
3) Stable text only if guaranteed invariant
4) Avoid CSS selectors tied to layout/styling

### 3.2 Prohibited
- `nth-child` selectors
- selectors dependent on DOM nesting depth
- classes generated by CSS-in-JS without stable contracts

## 4. Minimal UI Assertions (Recommended)

Any automated flow should assert:
- correct page/screen loaded
- primary CTA enabled/disabled as expected
- error state visible when invalid input
- success confirmation visible after completion
- network request fired (if applicable)

## 5. Environment Safety

If exposing internal state for testing, it must be:
- test-only (gated by env flag)
- not available in production builds
- never include secrets

## 6. Compliance

A UI change that breaks automation MUST:
- add missing test IDs / a11y labels, or
- update automation scripts accordingly

Non-compliant PRs may be blocked by review checklist.
